# Windows Event Log Analysis

## 1. Purpose

This document explains the most important Windows Security Event IDs for SOC monitoring, focusing on those relevant to the brute‑force detection scenario in this lab. Understanding these events allows a security analyst to detect, investigate, and respond to malicious activity without relying solely on SIEM alerts.

---

## 2. Event Log Basics

Windows logs are stored in `C:\Windows\System32\winevt\Logs\`. The most relevant log for security monitoring is the **Security** log.

- **Default size:** 128 MB (retains ~10–15 days of events on a busy system)
- **Access:** Event Viewer → Windows Logs → Security
- **Command line:** `wevtutil qe Security /f:text /c:100`

Wazuh forwards these events via the `eventchannel` log collector, preserving all fields.

---

## 3. Critical Event IDs for SOC Monitoring

| Event ID | Name | Typical Volume | SOC Relevance |
|----------|------|----------------|---------------|
| 4624 | Successful logon | High | Normal access – look for anomalies |
| 4625 | Failed logon | Low–Medium | **Brute‑force indicator** – core of this lab |
| 4634 | Logoff | High | Confirm session end |
| 4648 | Logon with explicit credentials | Medium | RunAs / service account usage |
| 4672 | Special privileges assigned | Medium | Admin logon – high value |
| 4698 | Scheduled task created | Low | Persistence mechanism |
| 4702 | Scheduled task updated | Low | Potential lateral movement |
| 4720 | User account created | Very low | Unauthorised account creation |
| 4726 | User account deleted | Very low | Suspicious cleanup |
| 4732 | Member added to security‑enabled local group | Low | Privilege escalation |
| 4740 | User account locked out | Low | Account lockout policy triggered |

## 4. Deep Dive: Event ID 4625 – Failed Logon

This is the most important event for detecting brute‑force attacks.

### 4.1 Event Structure (as seen in Wazuh)

```json
{
  "win": {
    "eventID": 4625,
    "eventData": {
      "subjectUserSid": "S-1-0-0",
      "subjectUserName": "-",
      "subjectDomainName": "-",
      "subjectLogonId": "0x0",
      "targetUserSid": "S-1-0-0",
      "targetUserName": "Administrator",
      "targetDomainName": "WIN-11",
      "status": "0xC000006D",
      "failureReason": "%%2313",
      "subStatus": "0xC000006A",
      "logonType": "10",
      "logonProcessName": "User32",
      "authenticationPackageName": "Negotiate",
      "workstationName": "WIN-11",
      "sourceNetworkAddress": "192.168.3.4",
      "sourcePort": "49234",
      "processId": "0x0",
      "processName": "-"
    }
  }
}
```

### 4.2 Key Fields for SOC Analysts

| Field | Meaning | Use Case |
|-------|---------|----------|
| `targetUserName` | Account being attempted | Identify targeted account |
| `sourceNetworkAddress` | IP address of attacker | **Essential for correlation & blocking** |
| `logonType` | Method of logon (3=network, 10=RDP) | Distinguish remote attacks from local mistakes |
| `status` / `subStatus` | Reason for failure | 0xC000006D = bad username or password; 0xC000006A = password wrong |
| `workstationName` | Source machine name (if available) | Identify compromised workstation |

### 4.3 Logon Type Reference

| Logon Type | Description | Typical Use |
|------------|-------------|-------------|
| 2 | Interactive | Console login |
| 3 | Network | SMB, file share |
| 10 | Remote Interactive | **RDP** – primary attack vector in this lab |
| 5 | Service | Service account startup |
| 7 | Unlock | Workstation unlock |

## 5. Brute‑Force Detection Logic

A brute‑force attack generates multiple 4625 events from the same source IP address within a short time window.

**Example query in Wazuh (using OpenSearch / Elasticsearch DSL):**

```json
{
  "query": {
    "bool": {
      "must": [
        { "term": { "win.eventID": 4625 } },
        { "range": { "@timestamp": { "gte": "now-60s" } } }
      ]
    }
  },
  "aggs": {
    "attackers": {
      "terms": { "field": "win.eventData.sourceNetworkAddress", "size": 5, "min_doc_count": 10 }
    }
  }
}
```

**Threshold used in this lab:**  
10 failures from the same source IP within 60 seconds triggers Wazuh rule 5710.

## 7. Correlation with Other Events

### 7.1 Successful Brute‑Force (If Attack Succeeded)

If the attacker eventually guesses the password, a **4624** event (successful logon) would appear:

```
Event ID 4624
Logon Type: 10
Target User: Administrator
Source IP: 192.168.3.4
```

**Analyst action:** Immediately flag the source IP for containment and assume compromise.

### 7.2 Account Lockout (Event ID 4740)

If the account lockout threshold is reached (e.g., 5 failures in 5 minutes), Windows generates **4740**:

```
Event ID 4740
Account: Administrator
Source IP: 192.168.3.4
Locked by: WIN-11
```

**Detection benefit:** Lockout events are also excellent brute‑force indicators and may trigger separate SIEM rules.

## 8. Wazuh Rule Customisation (Optional)

The default rule 5710 can be tuned to reduce false positives:

```xml
<!-- Example: require 20 failures in 120 seconds from same IP, exclude known admin console -->
<rule id="5710" level="12">
  <if_sid>5710</if_sid>
  <match>Windows EventID: 4625</match>
  <field name="win.eventData.sourceNetworkAddress" negate="no">\d+\.\d+\.\d+\.\d+</field>
  <field name="win.eventData.sourceNetworkAddress" negate="yes">127.0.0.1</field>
  <description>Multiple logon failures (10+ in 60s) from a single remote source</description>
</rule>
```

## 9. What to Look for Beyond Brute‑Force

| Suspicious Pattern | Event IDs | Indication |
|--------------------|-----------|------------|
| Account creation followed by privilege change | 4720, 4732 | Lateral movement / persistence |
| Scheduled task created by non‑admin process | 4698 | Malware persistence |
| Logon outside working hours | 4624, 4625 | Compromised credentials |
| Many logon failures then one success | 4625 → 4624 | Successful brute‑force |
| Event log cleared | 1102 (in System log) | Cover‑up attempt |

## 11. Summary for SOC Analysts

- **4625** is your primary signal for authentication‑based attacks.
- Always check **source IP** and **logon type** (type 10 = RDP = high risk).
- Use time‑based thresholds (e.g., 10 failures/60s) to reduce noise.
- Correlate with successful logons (4624) to determine if the attack succeeded.
- Combine with account lockout events (4740) for additional detection.
